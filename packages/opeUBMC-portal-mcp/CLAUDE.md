## 📝 Code Commit Documentation Requirements

### Mandatory Commit Records
Every code commit **MUST** include the following information:

#### 1. Commit Message Format
```
<type>: <subject>

<body>

Generated-By: <author> with Claude AI
Date: <YYYY-MM-DD HH:MM:SS>
Location: <file paths or module names>
Changes: <brief description of changes>

Co-Authored-By: Claude AI <noreply@anthropic.com>
```

#### 2. Commit Message Example
```
feat: Add user feedback management module

Implemented complete feedback CRUD operations including:
- Feedback creation API endpoint
- Feedback list with filtering and pagination
- Feedback statistics dashboard
- Admin feedback management interface

Generated-By: Zhang San with Claude AI
Date: 2026-02-06 14:30:00
Location: src/views/FeedbackManagement.vue, src/api/api-feedback.ts
Changes: Added 3 new files, modified 2 existing files, +450 lines

Co-Authored-By: Claude AI <noreply@anthropic.com>
```

#### 3. Required Information (5W)

**Who (谁)** - Developer and AI identification
**When (何时)** - Date and time (YYYY-MM-DD HH:MM:SS)
**Where (何地)** - File paths, modules, line changes
**What (何事)** - Type and description of changes
**Why (为何)** - Reason (optional but recommended)

#### 4. Commit Types
```
feat:     New feature
fix:      Bug fix
docs:     Documentation
refactor: Code refactoring
test:     Testing
chore:    Build/tools
security: Security fixes
perf:     Performance improvements
style:    Code formatting (no logic change)
```

### File-Level Documentation
For significant changes, add header comments:
```javascript
/**
 * @created 2026-02-06 by Zhang San with Claude AI
 * @modified 2026-02-06 by Zhang San with Claude AI
 * @description Brief description
 */
```

### Change Log (CHANGELOG.md)
Maintain for major features:
```markdown
## [1.2.0] - 2026-02-06
### Added
- Feature name
  - Generated by: Name with Claude AI
  - Files: path/to/files
  - Changes: +850 lines
```

### Git Commit Best Practices

1. **Atomic Commits**: One logical change per commit
2. **Descriptive Messages**: Clear and concise descriptions
3. **Reference Issues**: Link to issue tracker when applicable
4. **Sign Commits**: Use GPG signing for important commits
5. **Review Before Push**: Always review changes before pushing

## 🔌 MCP Server 开发规范

### 工具函数开发要求
- **MUST** 每个工具函数导出 `toolDefinition` 对象
- **MUST** 提供详细的中文 description（包括使用场景、参数说明、返回信息、示例问题）
- **MUST** 使用 JSON Schema 定义 inputSchema
- **MUST** 实现错误处理和超时控制（AbortSignal.timeout(15000)）
- **MUST** 实现缓存机制（15分钟）以减少 API 调用
- **MUST** 返回格式化的 Markdown 文本，不返回原始 JSON

### 工具函数命名规范
```javascript
// 文件名：camelCase
getSigInfo.js, getPackageInfo.js

// 函数名：camelCase
export async function getSigInfo() {}

// 工具名（toolDefinition.name）：snake_case
name: "get_sig_info"
```

### 工具函数结构模板
```javascript
// 数据源 URL（使用常量）
const BASE_URL = "https://api.openeuler.org/...";

// 缓存配置
let cachedData = null;
let cacheExpiry = 0;
const CACHE_DURATION = 15 * 60 * 1000; // 15分钟

// 主查询函数
export async function getToolName(param1, param2) {
  try {
    // 1. 检查缓存
    // 2. 发起 API 请求（带超时）
    // 3. 处理响应
    // 4. 格式化输出
    // 5. 更新缓存
  } catch (e) {
    if (e.name === "AbortError") {
      return "网络请求超时，请稍后重试。";
    }
    return `查询时发生错误：${e.message}`;
  }
}

// 工具定义
export const toolDefinition = {
  name: "get_tool_name",
  description: "详细的中文描述...",
  inputSchema: { /* JSON Schema */ }
};
```

### 工具函数测试要求
- **MUST** 测试正常查询场景
- **MUST** 测试空结果场景
- **MUST** 测试网络超时场景（15秒）
- **MUST** 测试缓存机制（验证缓存命中）
- **MUST** 测试参数验证（必需参数、可选参数）
- **MUST** 测试错误处理（API 错误、JSON 解析错误）

## 🌐 外部 API 集成规范

### API 调用标准
```javascript
// ✅ 正确示例
const response = await fetch(url, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(requestBody),
  signal: AbortSignal.timeout(15000), // 必须设置超时
});

if (!response.ok) {
  throw new Error(`API 请求失败：HTTP ${response.status}`);
}

const data = await response.json();
```

### 缓存策略
- **MUST** 为稳定数据实现缓存（如发行版生命周期、SIG 列表）
- **MUST** 设置合理的缓存过期时间（推荐 15 分钟）
- **MUST** 在缓存 key 中包含查询参数
- **MUST** 在缓存失效后重新获取数据

### 数据格式化规范
- **MUST** 统一使用 Markdown 格式输出
- **MUST** 限制列表显示数量（推荐 10 条，最多 20 条）
- **MUST** 在结果中说明总数和显示数量
- **MUST** 提供友好的错误消息（中文）
- **MUST** 对长文本进行截断（如描述超过 100 字符）

### 输出格式示例
```javascript
let result = "=== 标题 ===\n\n";
result += `共找到 ${total} 条记录，显示前 ${displayCount} 条\n\n`;
result += `1. 项目名称\n`;
result += `   描述: ...\n`;
result += `   版本: ...\n\n`;
return result;
```

## 🔒 Security Rules

### Input Validation
- **MUST** validate and filter all user inputs
- **MUST** use whitelist instead of blacklist
- **MUST** use parameterized queries (prevent SQL injection)
- **MUST** escape HTML content (prevent XSS)

### Sensitive Data
- **PROHIBITED** hardcoding passwords, keys, tokens in code
- **MUST** use environment variables for sensitive config
- **PROHIBITED** logging sensitive information
- **MUST** encrypt sensitive data in storage

### API Security
- **MUST** implement authentication and authorization
- **MUST** implement rate limiting
- **MUST** use HTTPS for sensitive data
- **MUST** validate API request origins

## 🧪 Testing Requirements

### Coverage Requirements
- **Each API endpoint: minimum 5 test cases**
  1. Normal scenario
  2. Boundary values
  3. Invalid inputs
  4. SQL injection test
  5. Permission test
- **Core functions: minimum 3 test cases**
- **Unit test coverage: ≥ 70%**

### Test Data
- **PROHIBITED** using production data for testing
- **MUST** use separate test database
- **MUST** clean up test data after tests

### Mandatory Test Code Policy

**CRITICAL RULE: No code commits without corresponding test code**

#### Enforcement Rules
1. **PROHIBITED** committing any new feature, bug fix, or refactoring without unit tests
2. **MANDATORY** test files must be included in the same commit as the implementation code
3. **Claude AI MUST automatically generate unit tests** if they are missing before commit
4. **Exceptions** (must be explicitly justified in commit message):
   - Documentation-only changes (docs:)
   - Configuration changes (chore:)
   - Style/formatting changes (style:)

#### Automatic Test Generation by Claude AI

When Claude AI detects missing tests, it **MUST**:

1. **Identify untested code**
   - New functions without corresponding test files
   - Modified functions without updated tests
   - API endpoints without test coverage

2. **Automatically generate comprehensive tests**
   - Create test file with proper naming convention (e.g., `user-service.test.js` for `user-service.js`)
   - Include all required test cases per coverage requirements
   - Follow project's testing framework and patterns
   - Add test data fixtures if needed

3. **Test file structure**
   - File header with creation date, author, and description
   - Setup and teardown sections for test initialization and cleanup
   - Organized test suites by module/function
   - Test cases covering all required scenarios:
     * Normal/happy path scenarios
     * Boundary value testing
     * Invalid input handling
     * Security testing (SQL injection, XSS, etc.)
     * Permission and authorization validation

4. **Commit message must include test information**
   ```
   feat: Add user authentication module

   Implementation includes:
   - User login endpoint
   - JWT token generation
   - Session management

   Tests included:
   - user-auth.test.js: 15 test cases
   - Coverage: 85% (functions), 90% (lines)

   Generated-By: [Author] with Claude AI
   Date: YYYY-MM-DD HH:MM:SS
   Location: src/auth/user-auth.js, tests/auth/user-auth.test.js
   Changes: +250 lines implementation, +180 lines tests

   Co-Authored-By: Claude AI <noreply@anthropic.com>
   ```

#### Pre-Commit Verification

Before any commit, Claude AI must verify:
- [ ] Implementation files have corresponding test files
- [ ] Test coverage meets minimum requirements (≥70%)
- [ ] All test cases pass successfully
- [ ] Test files follow naming conventions
- [ ] Tests cover all required scenarios (normal, boundary, error, security, permissions)

If tests are missing or insufficient, Claude AI **MUST**:
1. Generate the missing tests automatically
2. Run tests to ensure they pass
3. Include test files in the commit
4. Document test coverage in commit message

## 📝 Code Quality Standards

### Complexity Limits
- **Cyclomatic Complexity**: ≤ 10 per function, ≤ 50 per file
- **Nesting Depth**: ≤ 4 levels
- **Function Length**: ≤ 50 lines (excluding comments)
- **Function Parameters**: ≤ 4 parameters

### Naming Conventions
```javascript
// Files: camelCase (for MCP tools)
getSigInfo.js, getPackageInfo.js

// Functions/Variables: camelCase
function getUserById(), const userList

// Constants: UPPER_SNAKE_CASE
const MAX_RETRY_COUNT = 3
const CACHE_DURATION = 15 * 60 * 1000

// Booleans: is/has/should prefix
const isActive, hasPermission
```

### Error Handling
- **MUST** use unified error handling
- **MUST** not expose system error details to users
- **MUST** log errors with context

## 🚀 版本发布规范

### 语义化版本号规则
- **主版本号（Major）**: 不兼容的 API 变更（如修改工具名称、删除参数）
- **次版本号（Minor）**: 向下兼容的功能新增（如新增工具函数）
- **修订号（Patch）**: 向下兼容的问题修正（如修复 bug、优化性能）

### 发布前检查清单
- [ ] 所有测试通过
- [ ] README.md 已更新（工具列表、详细说明、项目结构）
- [ ] package.json 版本号已更新
- [ ] CHANGELOG.md 已更新（如果存在）
- [ ] 本地测试 MCP server 可正常启动（`npm start`）
- [ ] 使用 npx 测试包可正常安装和运行
- [ ] 所有新工具在 Claude Code 中测试通过

### 发布流程
```bash
# 1. 更新版本号
npm version patch  # 或 minor/major

# 2. 发布到 npm
npm publish

# 3. 推送 git tags
git push --tags

# 4. 创建 GitHub Release（可选）
gh release create v0.0.x --notes "Release notes"
```

### 版本号示例
- `0.0.1` → `0.0.2`: 修复 bug
- `0.0.2` → `0.1.0`: 新增工具函数
- `0.1.0` → `1.0.0`: 重大 API 变更

## 🚫 Prohibited Practices

### Absolutely Forbidden
1. ❌ Using unauthorized or infringing code
2. ❌ Committing code with passwords/keys
3. ❌ Direct SQL string concatenation
4. ❌ Outputting user input to HTML without escaping
5. ❌ Logging sensitive info in production
6. ❌ **Submitting code without tests (Claude AI must auto-generate if missing)**
7. ❌ Using `any` type (TypeScript) without justification

### openEuler Portal MCP 特定禁止项
1. ❌ 不要在工具描述中使用模糊的术语（必须明确说明使用场景）
2. ❌ 不要返回未格式化的 JSON 数据给用户（必须格式化为 Markdown）
3. ❌ 不要在单次查询中返回超过 1000 条记录（建议 10-20 条）
4. ❌ 不要硬编码 API 端点（应使用常量 BASE_URL）
5. ❌ 不要忽略 API 返回的错误码（必须检查 response.ok 和 data.code）
6. ❌ 不要在工具描述中混淆"技术 SIG"和"治理委员会"
7. ❌ 不要在没有缓存的情况下频繁调用外部 API
8. ❌ 不要在 fetch 请求中省略超时设置（AbortSignal.timeout）
9. ❌ 不要在错误消息中暴露内部 API 地址或敏感信息
10. ❌ 不要使用同步阻塞操作（必须使用 async/await）

### Not Recommended
1. ⚠️ Excessive nested ternary operators
2. ⚠️ Magic numbers (use named constants)
3. ⚠️ Overly long functions or files
4. ⚠️ Global variables (except necessary config)

## 📚 Performance Guidelines

### Database
- Avoid N+1 queries
- Use indexes for optimization
- Avoid `SELECT *`
- Paginate large datasets

### Backend Services
- Implement connection pooling for databases and external services
- Use caching strategies (Redis, in-memory) for frequently accessed data
- Implement request rate limiting and throttling
- Use asynchronous processing for long-running tasks
- Optimize API response payloads (return only necessary fields)
- Implement proper timeout handling for external API calls
- Use batch processing for bulk operations
- Monitor and log performance metrics
- Implement circuit breakers for external service calls

### MCP Server Specific
- **MUST** implement caching for stable data (15-minute cache)
- **MUST** set request timeouts (15 seconds recommended)
- **MUST** limit result set sizes (10-20 items per query)
- **MUST** use async/await for all I/O operations
- **MUST** handle network errors gracefully
- **MUST** provide progress feedback for long operations
- **SHOULD** implement retry logic for transient failures
- **SHOULD** batch multiple related queries when possible
